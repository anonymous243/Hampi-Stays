# Railway Secure Deployment Notes for HampiStays

To maintain the enterprise-grade security established in the codebase, follow these steps during your Railway deployment:

### 1. Environment Variables Configuration
Ensure ALL of the following are set in the Railway dashboard. Never use "Public" visibility for sensitive keys.

- **`ENCRYPTION_KEY`**: Generate a 32-byte hex string. 
  - *Command*: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
- **`JWT_SECRET`**: Generate a long, random string.
- **`NODE_ENV`**: Set to `production`.
- **`DATABASE_URL`**: Provided by Railway PostgreSQL. Ensure it includes `sslmode=require`.
- **`FRONTEND_URL`**: Set to your custom domain or Railway `.app` URL.

### 2. Database Hardening
- Railway PostgreSQL handles SSL by default. Ensure your `DATABASE_URL` uses it.
- Run `npx prisma migrate deploy` in your build step to apply schema changes securely.

### 3. Monitoring
- Check Railway logs for `[SECURITY ALERT]` tags generated by our `errorHandler` middleware.
- Monitor for unusually high rate-limit hits (429 errors).

### 4. Secret Rotation
- It is recommended to rotate `JWT_SECRET` and `ENCRYPTION_KEY` every 90 days.
- Note: Rotating `ENCRYPTION_KEY` will require a data migration script to re-encrypt existing data. (Utilities for this are planned for Phase 2).

### 5. Dependency Audit
- We have performed an initial scan. Run `npm audit` periodically on your local machine before pushing to GitHub.
